## Problem

The screen-share permission prompt appears every time sync starts. Root cause (confirmed in code):

1. **No persistent authorization is requested.** `CreateSession` (`capture.py:231-233`) passes only `session_handle_token` — no `persist_mode` — so the portal never issues a `restore_token`. The `restore_token` returned in the `Start` response is discarded (only `streams` is read, `capture.py:224-226`).
2. **The session is torn down on every stop.** `SyncController.stop()` → `capture.stop_pipeline()` → `_close_portal_session()` + clears `_portal_node_id` (`sync.py:193-195`, `capture.py:633-642`). With `reading_mode.auto_activate=true` by default, every stop→start cycle rebuilds the session and re-prompts.

Per your choices: **release the session on stop** (so the desktop's screen-share indicator clears — current stop behavior stays as-is), but **remember consent forever** across app restarts using the portal's `persist_mode=2` + `restore_token` mechanism.

## Fix

The entire behavior change lives in how `_setup_portal_session()` talks to the portal, plus plumbing to persist the token. Stop/release logic is unchanged.

### 1. `src/lumux/config/settings_manager.py` — add token field to `CaptureSettings`
```python
@dataclass
class CaptureSettings:
    scale_factor: float = 0.125
    source_type: str = "screen"
    restore_token: str = ""  # Portal restore token for persistent screen-cast consent
```
No load/save code changes needed — `_load_settings` already does `CaptureSettings(**data.get("capture", {}))` and `save()` already does `asdict(self._settings.capture)`, so the new field flows through automatically.

### 2. `data/default_settings.json` — add default
Add `"restore_token": ""` to the `capture` object (keeps the shipped defaults file in sync with the schema).

### 3. `src/lumux/capture.py` — request persistence, capture & expose the token
- **`__init__`**: accept `restore_token: str = ""`, store `self._restore_token`; init `self._on_restore_token_changed = None`.
- **New method** `set_on_restore_token_changed(callback)` and **property** `restore_token` (read access for diagnostics).
- **`_setup_portal_session()`**:
  - Build `CreateSession` options with `"persist_mode": GLib.Variant("u", 2)` (persist until explicitly revoked) plus `"restore_token": GLib.Variant("s", self._restore_token)` **only if** a stored token exists. Passing an empty token string can confuse some backends, so it's conditional.
  - In the `on_response` handler's `elif "streams" in results:` branch (Start response), also read `results.get("restore_token", "")` and `results.get("persist_mode", 0)` into `state`.
  - After `Start` succeeds (`capture.py:292-295`): if the returned token differs from the stored one, update `self._restore_token` and invoke the `on_restore_token_changed` callback (wrapped in try/except so a persistence failure never breaks capture). This also clears a stale token if the portal didn't grant persistence.
- **Stale-token retry** at the call site in `capture()` (`capture.py:144-146`): if `_setup_portal_session()` fails **and** we had a stored token, clear `self._restore_token` and retry once. Handles the common case of a monitor/source change invalidating the token — without this, a stale token would cause a silent capture stall.

### 4. `src/lumux/app_context.py` — wire persistence
- Pass `restore_token=settings.capture.restore_token` into the `ScreenCapture(...)` constructor (`app_context.py:32-36`).
- Register a callback that writes the token back to settings and saves:
```python
def _on_restore_token_changed(self, token: str):
    self.settings.capture.restore_token = token
    try:
        self.settings.save()
    except Exception as e:
        timed_print(f"Error persisting capture restore token: {e}")

self.capture.set_on_restore_token_changed(self._on_restore_token_changed)
```

## Resulting behavior
- **First sync ever**: prompt shown (unavoidable). GNOME 47+ (runtime is 50) grants persistence → token stored to `settings.json`.
- **Every subsequent sync** (same app run or after restart): portal restores silently from the token, **no prompt**. Session is still released on stop, so the screen-share indicator clears.
- **Monitor/source change**: stale token → setup fails → automatic one-time retry without token → fresh prompt → new token stored.

## Notes / out of scope
- Portal persistence requires a supporting backend (GNOME 47+, KDE Plasma 6.x). On older backends that ignore `persist_mode`, no token is returned and the prompt appears each time — this is a desktop-environment limitation, not a bug we can fix in-app.
- No change to `stop_pipeline()` / the release-on-stop behavior, per your choice.
- No tests in the repo for `capture.py` (it requires a live portal/display), so no test changes to make; manual verification is the path here.
